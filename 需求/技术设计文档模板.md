## 1、需求背景（必须）

_简述项目/需求的背景，如项目目标、项目方案、产品功能现状和本期需求变更情况，本次变更预期的收益等等。_

_建议粘贴需求wiki地址即可。_

## 2、需求理解（必须）

_1、RD角度需求理解_

_RD角度对PM同学提出的需求的理解，不是复述，而是总结和抽象；对需求进行拆解，映射到功能层面；作为流程、接口定义的依据；_

_描述角度：新增了哪些功能，已有的功能做了哪些变更_

_2、引申需求_

_除了原始需求外，可能会涉及历史遗留的逻辑问题，或优化类的需求（响应耗时、实时返回调整为异步处理等），不属于PM同学所提的需求范围，但对需求的实现存在影响，需要纳入考虑。_

_3、内容要求_

_能够让参与评审的同学（PM、非该模块负责的RD、项目QA），基本明确需求影响的功能模块，预期实现的目标，和影响范围。_

_总结：列举本次需求影响面拆解，列举功能改造点，包括旧数据改造等，比如新增了XXX功能，更新了XXX（from...to....）、删除了XXX，刷数据等_

## 3、 概要设计

### 3.0 设计思路与折衷（可选）

_简要描述本次需求的设计思路如：设计的关键模块/关键功能的实现思路、可能遇到的性能瓶颈与解决方案、可能对其他系统的影响及解决方案、模块的划分思路及边界等_

_总结：常规性迭代可不写，大的功能性改造建议描述下，特别涉及架构修改的，新引入功能的，比如：作业中心父子流程设计等。_

### 3.1 系统架构（必须，可重复使用）

_本次需求/项目涉及模块的系统关系图和架构图，需要描述清楚：1. 本系统与外部系统的依赖关系和交互；2. 本系统内部的模块划分、依赖关系和交互_

_总结：此处描述清楚依赖的上下游关系，需要交互的系统及对应RD等；不一定画架构图。能够让大家清晰本次需求和外部的交互即可；_

### 3.2 接口列表（必须）

_本次需求/项目涉及的所有新增的和变更的接口列表，需要将接口放在落兵台 （[http://weapons.ke.com](http://weapons.ke.com/project/1265/wiki/page/)）_

_接口列表的书写格式：_

_1. 按照需求页面的展现逻辑来分类，如：先列出需求的页面截图，再描述实现需求（包括信息展示类接口、提交类接口、功能交互类接口、模板页面的数据接口等）需要什么接口，再一一列出接口的列表_

_2. 按照某个独立功能的实现流程来分类，如：先列出某个功能的流程图，在流程图中体现出接口的列表，再一一列出其中的接口信息_

_总结：每个接口必须包含：接口描述、接口路径、接口类型（GET/POST）、全部接口参数（包含每个参数的含义）、接口正常返回格式、接口异常返回格式、所有返回值的类型和枚举值以及字段含义；此处应详细编写，也是**该文档的重点**；_

## 4. 详细设计

### 4.1 流程图（非必须）

_以概要设计中的接口定义为基础，绘制每个接口的流程图，流程图绘制方法可参考：[https://zhuanlan.zhihu.com/p/67533900](https://zhuanlan.zhihu.com/p/67533900)_

_流程图中，查询逻辑需要标明查询参数、判断逻辑需要标明判断条件、循环逻辑需要在流程图中明示_

_流程图中未描述完成的逻辑，可在流程下方辅以文字补充_

_对于存在两个及以上判断或循环逻辑的流程，需要增加伪代码描述，伪代码的书写方法可参考：[https://zh.wikihow.com/%E7%BC%96%E5%86%99%E4%BC%AA%E4%BB%A3%E7%A0%81](https://zh.wikihow.com/%E7%BC%96%E5%86%99%E4%BC%AA%E4%BB%A3%E7%A0%81)_

_总结：复杂的接口交互，建议在这儿画下时序图；_

### 4.2 数据表设计（非必须）

_可以以 4.1 的流程图为组织结构，增加流程涉及的表结构定义，也可以以数据模型设计为基础描述表结构设计。_

_若为本次新增的表，需要写明表名、表注释、表字段、表索引结构，可以用建表语句描述（建表语句需要符合公司 [mysql 规范](http://docs.intra.ke.com/dba/beike_mysql_standard.html)__），也可以用表格描述_

_所有的表字段需要注明：字段名、字段含义、字段枚举值、字段类型、字段长度范围、字段取值来源、与其他表的关联关系（若无关系可不填）_

_默认通用字段规范:（可根据实际情况调整）_

`id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '主键',

`creator` bigint(20) NOT NULL DEFAULT '0' COMMENT '创建人',

`updater` bigint(20) NOT NULL DEFAULT '0' COMMENT '更新人',  
`create_time` datetime NOT NULL DEFAULT '1970-01-02 00:00:00' COMMENT '创建时间',  
`update_time` datetime NOT NULL DEFAULT '1970-01-02 00:00:00' COMMENT '更新时间',

`delete_time` datetime NOT NULL DEFAULT '1970-01-02 00:00:00' COMMENT '软删除时间',  
`memo` varchar(256) NOT NULL DEFAULT '' COMMENT '备注',  
`is_delete` tinyint(4) NOT NULL DEFAULT '0' COMMENT '逻辑删除标志',

PRIMARY KEY (`id`) USING BTREE,

_总结：如有数据库更新，建议此处详细写清楚变更范围，比如新增表，新增列，更新的索引等；_

#### _4.2.1 数据库中敏感字段加密方案说明（非必须）_

_若数据库表中涉及到C3/C4敏感字段，需要进行加密存储，C3/C4标准[3.涉密数据申请规范（数据安全）](https://wiki.lianjia.com/pages/viewpage.action?pageId=601012791)，加密算法标准：[数据库统一加解密方案](https://wiki.lianjia.com/pages/viewpage.action?pageId=590082835)_

_总结：如有就描述，没有就算了。也可以放在上面统一说；_

#### _4.3 数据规模与性能评估（非必须）_

_本项目/需求涉及功能的数据规模和性能评估_

_如列表类页面，涉及的数据查询量和接口性能预估；提交类页面涉及的提交接口性能预估；处理逻辑中涉及的外部系统调用接口查询出来的数据规模预估和处理逻辑的性能预估等_

_涉及定时/一次性执行脚本，在评估数据规模后及系统性能后，设置合理的间隔时间，并将脚本配置书写在技术方案中_

_备注：对于新的产品形态无法准确预估数据规模的，可以按照一定的推导逻辑推导出数据量，对推导结果的准确性不做要求，但是对推导逻辑及基于推导结论作出的性能评估要求准确且合理_

_总结：此处可以列举数据库风险，比如数据规模，一般没有哈。_

### 4.4 特殊功能设计（可选）

_本次设计中的特殊功能描述：如配置项及开关逻辑、系统迁移的平滑上线方案、大数据量的导出处理、进程管理或 kafka 等特殊使用场景_

_总结：此处列举apollo配置，开关逻辑，kafak配置等；_

### 4.5 上线步骤（必须）

_本项目/需求的上线步骤，按照操作上线的实际所需步骤进行罗列，每操作一个系统/执行一条命令为一条独立的步骤。_

_每条步骤都必须包含：预期操作时间、操作人、操作的系统/执行的命令、预期的执行结果、线上验证方案、回滚方案、验证人、验证结果_

_需要操作的系统和命令需要写清楚操作路径和操作命令，如：代码上线要附上对应的项目 git 地址，sql 上线要写清楚具体 sql 或对应的 sql 文件、定时任务要写清楚定时任务平台地址和任务 id/任务名称以及任务命令全文（新增情况下）_

_验证方案需要附带验证的操作步骤和验证需要的数据，如：功能的 url，待验证的需要录入的数据准备、待验证的展示内容、日志的关键词和fast中的事件名、需要在 databus 验证的数据库名和 sql 及数据_

_所有的验证结果需附带对应的截图_

_无法回滚的需要标明无法回滚的原因及出现问题时的解决预案（如周知到什么人、做什么系统操作、在哪里做什么通报等等）_

_总结：此处明确本需求，涉及的服务上线注意事项，是否是基础服务，是否可以提前上线等_

### 4.6 数据迁移（非必须）

_明确业务原有作业场景工具，并确定数据迁移到新系统的方案。_

_总结：此处描述刷数据，迁移数据方案，需要提供的工具等_

_技术上的不确定性，潜在的人力、时间风险，存在需求通知上下游的变更等；_

_方案变更、上下游通知必须正式的邮件通知，并确保相关各方已经明确变更内容 和 截止时间点_

_总结：本次需求，有无风险，什么风险，有无兜底策略；_
