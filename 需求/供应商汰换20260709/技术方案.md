## 1、背景

PRD：https://wiki.lianjia.com/pages/viewpage.action?pageId=1577332421 Ones：https://ones.ke.com/project/requirement/50545697

开发分支：feat/50545697/replacesupplier

项目生成后基于"品类-供应商"生成货前服务任务，货后流转中可能发生供应商汰换。汰换由上游发起，明确告知汰换方式：**常规汰换 / 紧急汰换+立即切换 / 紧急汰换+不立即切换**。任务侧需结合汰换方式 + 新老商角色 + 老商任务状态进行处理。

项目生成后，系统基于"品类-供应商"配置生成测量、复尺、设计、报价四类货前服务任务。在货后流转中，上游可能发起供应商汰换——将某个 SKU×店铺组合的供应商从 A 换成 B。汰换有三种方式：

|   |   |   |
|---|---|---|
|类型|编码|含义|
|常规汰换|1|正常替换，参考已有货单状态决定履约方|
|紧急+立即|2|紧急替换且立即生效，需同步取消 OMS 货单|
|紧急+不立即|3|紧急替换但当前货单继续由老商完成|

### 2.4 核心处理逻辑

#### 阶段 1：消息接收

上游（商品/供应链系统）通过 MQ 发送 `MerchantSkuSupplierReplacePayload` 消息，包含：SKU ID、店铺 ID、老供应商编码/名称、新供应商编码/名称、品类编码/名称、汰换类型、是否立即切换、变更时间戳。

starlord 接收后调用 `SupplierReplaceMessageServiceImpl.save()`，将消息转为 `SupplierReplaceMessage` 记录落表，状态标记为 `PENDING`（待处理）。

> _💡 同一_ _SKU__×店铺在一天内可能发生多次汰换（A→B→C→D），每条变更独立记录，由 T+1_ _批处理__统一聚合。_

#### 阶段 2：T+1 批量汇总 → 发需求层

定时任务每天触发 `SupplierReplaceScheduleServiceImpl.processT1Batch()`，执行四步：

Step 1 — 捞取待处理消息 查询 `supplier_replace_message` 表中昨日 `status=PENDING` 的全部记录。

Step 2 — 按 SKU×店铺聚合 `SupplierReplaceAggregateService.aggregate()` 将所有消息按 `skuId+merchantId` 分组，每组内按变更时间升序排列，取最早记录的 oldSupplier 与最晚记录的 newSupplier 作为该组的最终汰换态。

Step 3 — 履约判断 对每个聚合组，通过 `SupplierReplaceHiveManager` 查询 Hive 数据仓库，获取该 SKU×店铺下所有有用量的项目订单及其货单信息（订单ID、货单号、货单状态、供应商编码）。

然后调用 `SupplierFulfillJudgeService.judge()` 按汰换类型判定履约方：

- 常规汰换：查该订单下老商的任务，若存在已完成且已同步 SDM（syncToSdm∈{1,2}）→ 老商履约；否则新商履约
    
- 紧急+立即：无货单 → 新商履约；货单状态 ≤ 2600（未发货）→ 新商履约；货单已发货 → 老商履约
    
- 紧急+不立即：无货单 → 新商履约；有货单 → 老商履约
    

Step 4 — 按订单汇总发送需求层 将同一 `projectOrderId` 下所有 SKU×店铺组合的履约结果汇总为一个 `SupplierReplacementParam`，调用 `DemandLayerManager.sendReplaceDemand()` → `ReceiptOrderApi.handleReceiptServiceOrderSupplierReplacement()` 发送给需求层。发送成功后消息状态标记为 `PROCESSING`。

#### 阶段 3：需求层回调 → starlord 执行任务操作

需求层处理完成后，通过内部事件/Kafka 回调 `SupplierReplaceDemandEventHandler`，消息体为 `SupplierReplaceDemandPayload`（含 projectOrderId、品类、老商/新商编码、履约结果、汰换类型等）。

`SupplierReplaceProcessServiceImpl.process()` 执行以下步骤：

3.0 — 主材申请单 触发 `scmMeasureApplyEventHandler.handleBiz()`，生成主材申请单任务。

3.1 — 老商履约场景

如果履约结果为"老商履约"，则对新老商任务进行比对：按 taskType（测量/复尺/设计/报价）逐类型判断，若该 taskType 的新老商均非供应商配置任务（即都是内部履约角色），则取消老商任务，流程结束。

3.3 — 激活新商任务 若新商任务存在且 `planActivateTime < now`，调用 `materialActivateV2Service.activateTaskDispatch()` 激活任务。

3.4 — a 场景处理（均非供应商）

若某个 taskType 的新商和老商任务都不是供应商配置任务（syncToSdm=0 且配置中无供应商角色），则复制老商任务状态到新商任务：

- Task 级：复制 orderNo、processStatus、currentNodeType、syncToSdm
    
- Node 级：按 nodeType 匹配，将老商每个节点的执行人、时间、状态等全部覆写到新商对应节点
    

  

3.5 — 并行后处理

- 货主材 + 考核：调用 `materialActivateV2Service.activityDeliveryFlowTaskDispatch()`
    
- 排程：调用 `DeliveryProcessFeignService#demandChange`（⚠️ 当前未接入，为 TODO）
    

3.6 — OMS 货单取消重下 仅 `replaceType=2`（紧急+立即汰换）时触发：调用 `FulfillmentApi#supplierReplace`，OMS 先同步取消老货单及货后任务，再异步下新货单。

#### 关键业务规则总结

|   |   |
|---|---|
|规则|说明|
|聚合维度|skuId + merchantId，T+1 批处理|
|履约判断|3 种汰换类型对应 3 套判断逻辑，核心看"老商货单是否已发货 / 任务是否已完成并同步 SDM"|
|任务取消|老商履约 + b 场景（含供应商任务）→ 取消老商未完成任务|
|状态复制|a 场景（均非供应商任务）→ 老商任务状态全量覆写到新商，保证连续性|
|OMS 联动|仅紧急+立即汰换触发 OMS 取消重下，其他方式不动货单|
|排程通知|所有场景都需通知排程侧供应商变更（当前未接入）|

  

#### 角色场景判断

如果新商任务和老商任务都不是供应商任务，则a场景，

b场景 && 老商任务，才取消

```Plain
老商角色：有老商任务 → 看任务角色；无 → 看老商配置；都没有 → 归属 b
新商角色：看新商配置；无配置 → 归属 b
两者都可判断 → 区分 a/b；任一不可判断 → 默认 b
```
