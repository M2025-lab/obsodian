

# 一、整体业务流程（完整家装主材流程）

整个流程：

```text
客户签约
    │
    ▼
意向报价
    │
    ▼
设计方案
    │
    ▼
主材申请单
    │
    ▼
货前任务（Pre-Goods）
    │
    ├──测量（可选）
    ├──复尺（二次量尺）
    ├──设计
    └──报价
    │
    ▼
确认下单
═══════════════
货物正式产生（货前→货后分界点）
═══════════════
    │
    ▼
货后任务（Fulfillment）
    ├──下单
    ├──接单
    ├──备货
    ├──排产（部分品类）
    ├──发货
    ├──配送
    ├──进场
    ├──安装
    └──完工
```

这里最重要的一点：

> **"是否已经产生货物（是否下单）" 是货前和货后的唯一业务分界。**

不是安装，不是设计，而是：

> **正式下单 = 开始履约 = 进入货后阶段。**

---

# 二、货前（Pre-Goods）业务

货前所有动作都有一个共同特点：

> **没有真正产生货物，只是在确认需求。**

包括：

```
签约
↓

初步报价（意向报价）

↓

设计方案

↓

测量（部分场景）

↓

复尺（二次量尺）

↓

正式报价

↓

等待确认下单
```

里强调：

现在很多品类：

> 测量已经没有了

只保留：

> 复尺（复尺=二次量尺）

因为：

第一次量尺很多已经提前完成。

---

## 为什么叫货前？

举例：

客户想安装木门：

```
客户说：
我要一个木门。

↓

设计师画图

↓

量尺寸

↓

复尺

↓

报价

↓

客户确认
```

直到：

```
厂家收到订单
```

之前：

```
世界上还没有这扇门。
```

所以：

都是货前。

---

# 三、货后（Fulfillment）

一旦下单：

厂家开始生产：

```
木门开始存在
```

所以进入：

货后。

里列举：

```
下单

↓

接单

↓

备货

↓

排产（部分品牌）

↓

发货

↓

运输

↓

进场

↓

安装

↓

完成
```

这些统称：

**货物履约流程（Fulfillment）**

它关注的是：

> 一件货如何最终送到客户家。

---

# 四、货前和货后的关联关系

里面有一句最重要的话：

> **货后任务通过货单号关联货前任务。**

可以画成：

```text
货前任务

测量
复尺
设计
报价

        │
        │生成需求
        ▼

      下单

        │

货单号(Order No)

        │

        ▼

货后任务

接单

备货

发货

安装
```

所以：

货前没有货单号。

货后一定有货单号。

---

# 五、主材申请单（最核心概念）

反复解释：

很多新人都会问：

> 主材申请单到底哪里来的？

正确流程应该是：

```
客户签约

↓

初步报价

↓

设计图

↓

设计师创建主材申请单

↓

进入主材系统
```

所以：

**主材申请单不是本系统创建的。**

它来自：

前置业务系统。

明确说：

> 触发点不在我们这。

---

## 主材申请单作用

它实际上代表：

> 这户客户需要哪些材料。

例如：

```
客户：

卫生间：

索菲亚木门

卧室：

金属门

厨房：

玻璃门
```

这些都会记录在：

主材申请单。

---

它其实就是：

> 后续所有任务的起点。

---

# 六、需求层（Requirement Layer）

里面：

需求层其实就是一个名字。

真正含义：

```
需求层

=

主材申请单

+

测量申请单

+

其它需求单据
```

也就是说：

需求层

不是一个新业务。

只是：

业务人员起的名字。

---

# 七、SKU是什么

举了京东的例子。

例如：

百事可乐：

```
330ml

330ml×6

500ml

500ml×12
```

虽然都是：

百事可乐。

但是：

SKU不同。

---

家装也是一样：

木门：

```
品牌：

索菲亚

规格：

2100×900

颜色：

白色

材质：

实木

左开门
```

这就是：

一个SKU。

换颜色：

又一个SKU。

换尺寸：

又一个SKU。

---

所以：

主材申请单：

最后都会拆成：

SKU。

---

# 八、SKU拆分流程

说：

```
主材申请单

↓

SKU

↓

拆分

↓

形成具体任务
```

例如：

一个客户：

```
木门

玻璃门

卫浴

门锁
```

实际上：

会拆成：

```
SKU1

SKU2

SKU3

SKU4
```

每一个SKU：

都会走自己的履约流程。

---

# 九、任务（Task）的真正作用

讲得很重要：

很多任务：

其实只是：

> 载体。

例如：

设计任务：

不是：

没有设计任务：

设计师就不能设计。

真正意义：

```
设计任务

=

记录：

什么时候开始

什么时候完成

谁负责
```

同理：

报价任务：

不是：

没有任务：

不能报价。

而是：

```
报价任务

=

报价行为的记录
```

因此：

任务更多承担：

流程管理。

不是：

业务本身。

# 十一、供应商变更为什么复杂

后半部分主要讲的是：

商品系统。

例如：

原来：

```
SKU

↓

供应商A（索菲亚）
```

后来：

改成：

```
SKU

↓

供应商B（欧派）
```

商品系统：

只知道：

```
SKU换供应商
```

但是：

不知道：

到底影响：

哪些订单。

因此：

当前系统：

收到消息后：

还要：

```
SKU

↓

查询数据库

↓

找到所有受影响订单

↓

更新任务

↓

更新履约关系
```

所以：

说：

商品消息：

只是：

告诉你：

SKU变了。

真正影响：

需要业务系统自己查。

---

# 十二、为什么需要聚合（Aggregation）

举了一个经典例子：

SKU：

```
A

↓

B

↓

C

↓

A
```

如果：

每一次：

都处理。

系统就会：

```
A→B

处理

↓

B→C

处理

↓

C→A

处理
```

实际上：

最终：

还是：

A。

等于：

没有变化。

所以：

系统会：

```
消息落库

↓

聚合

↓

计算最终状态

↓

只有真正变化

才处理
```

这样：

可以避免：

大量无效任务。

---

# 十三、系统数据流

最后描述的数据流可以整理成：

```text
商品中心
        │
        │ SKU供应商变更消息
        ▼
消息表（Message）
        │
        ▼
中间表（落库）
        │
        ▼
聚合处理
        │
        ▼
查询HAVE业务数据
        │
        ▼
找到受影响订单
        │
        ▼
判断：
    是否供应商切换
    是否内部角色变化
    是否需要创建/迁移任务
        │
        ▼
更新货前/货后任务
```

# 十四、核心概念总结

| 概念    | 含义                      | 是否本系统创建   |
| ----- | ----------------------- | --------- |
| 签约    | 客户确认装修意向                | 否         |
| 初步报价  | 意向报价，用于确认需求             | 否         |
| 设计    | 输出设计方案和图纸               | 否（任务可记录）  |
| 主材申请单 | 描述客户需要哪些主材，是后续流程的起点     | 否（前置系统生成） |
| 需求层   | 主材申请单、测量申请单等需求单据的统称     | 否         |
| 测量    | 首次量尺，部分场景保留             | 是货前任务     |
| 复尺    | 二次量尺，确认最终生产尺寸           | 是货前任务     |
| 报价任务  | 报价行为的流程载体，不等同于报价业务本身    | 是货前任务     |
| SKU   | 唯一物料规格（品牌+规格+材质+颜色等）    | 来自主材申请单拆分 |
| 下单    | 履约正式开始，也是货前与货后的分界点      | 是         |
| 货单号   | 履约订单唯一标识，用于串联货后流程       | 是         |
| 货前任务  | 下单前的需求确认任务，如测量、复尺、设计、报价 | 本系统管理     |
| 货后任务  | 下单后的履约任务，如接单、备货、发货、安装等  | 本系统管理     |
| 聚合处理  | 合并多次SKU/供应商变更，仅处理最终有效状态 | 本系统实现     |
| 消息表   | 接收上游商品变更消息的落库表          | 本系统维护     |
| 中间表   | 保存聚合及查询结果，供后续业务处理       | 本系统维护     |

---

## 业务主线（一句话概括）

整个家装主材系统的本质可以概括为：

> **以前置系统生成的主材申请单作为业务起点，将客户主材需求拆分为具体 SKU，通过货前任务（需求确认）完成下单准备；以下单作为货前与货后的业务分界，以货单号串联供应商履约全过程（接单、备货、发货、安装等），同时通过商品变更消息、数据聚合和订单关联机制，保证供应商、SKU 与履约任务的一致性。**
