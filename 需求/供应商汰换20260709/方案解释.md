
整个技术方案其实只有一句话：

> **供应商发生汰换后，Starlord先收集所有数据并判断最终应该由谁履约，然后交给需求层修改施工需求，需求层完成后再通知Starlord生成/取消任务，最后同步OMS、排程等外围系统。**

整个流程其实可以拆成 **五个阶段**。

这样别人第一次看就能理解。

---

# 第一章：为什么要做（业务背景）

先回答三个问题。

## 什么是供应商汰换？

例如：

```
订单1001

橱柜

供应商A
```

后来：

```
供应商A
↓

供应商B
```

就是供应商汰换。

但是问题来了：

A可能：

- 已经测量
    
- 已经设计
    
- 已经下货
    
- 已经发货
    

不同阶段：

处理方式完全不同。

所以：

系统不能简单把供应商改成B。

必须判断：

**到底继续让A履约？**

还是：

**切给B履约？**

---

## 为什么复杂？

因为存在三种汰换：

```
① 常规汰换

② 紧急立即切换

③ 紧急但不立即切换
```

不同类型：

处理规则完全不同。

例如：

```
已经发货

↓

不能切B

只能继续A
```

而：

```
还没发货

↓

可以直接切B
```

所以：

需要系统统一判断。

---

## 最终目标

一句话：

> **根据汰换类型、货单状态、历史任务状态，计算最终履约方，并保证所有业务系统保持一致。**

---

```
              上游商品系统
                    │

                    │
                    ▼
              Starlord
        （汇总 + 判断履约）

      ▼      ▼        ▼
    OMS     排程     考核
```

这张图比现在文档里的各种图更容易理解。

# 第三章：整个业务流程（重点）

这一章建议作为整篇文档重点。

不要直接讲代码。

直接讲业务。

整个流程分五步。

---

## 第一步：接收供应商汰换消息

上游发送：

```
A

↓

B
```

消息包含：

```
sku

店铺

老供应商

新供应商

时间

汰换类型
```

Starlord：

先全部落表。

为什么？

因为：

一天可能：

```
A→B

B→C

C→D
```

不能立即处理。

否则：

一天会改三遍任务。

所以：

先全部保存。

---

## 第二步：T+1聚合

第二天：

定时任务开始工作。

例如：

数据库：

```
A→B

B→C

C→D
```

聚合以后：

变成：

```
A→D
```

这样：

后面：

所有系统：

只改一次。

这一步是整个方案最大的优化点。

建议重点强调。

---

## 第三步：判断最终谁履约（核心）

这是整个方案最重要的地方。

建议单独一章。

这里只做一件事：

判断：

```
老商？

还是新商？
```

规则：

### 常规汰换

看：

```
老商任务

↓

是否已经完成

↓

是否同步SDM
```

如果：

完成：

```
老商继续
```

否则：

```
新商
```

---

### 紧急立即

看：

```
有没有货单？
```

没有：

```
新商
```

有：

继续看：

```
货单是否发货
```

没发：

```
新商
```

已发：

```
老商
```

---

### 紧急不立即

更简单：

```
有货单

↓

老商
```

```
没货单

↓

新商
```

建议这里画三个流程图。

阅读体验会非常好。

---

# 第四步：通知需求层

很多人不知道需求层干什么。

所以建议单独介绍。

可以这样写：

需求层负责维护：

```
施工需求
```

它是整个施工流程的数据源。

所以：

Starlord：

只是告诉需求层：

```
这个订单

最终：

应该由谁履约。
```

需求层：

修改：

```
ReceiptOrder
```

然后：

发送：

```
SupplierReplaceDemandEvent
```

回来。

这一步：

职责一下就清楚了。

---

# 第五步：Starlord执行真正的任务变更

收到需求层事件以后。

开始真正修改任务。

依次：

```
取消老任务

↓

生成新任务

↓

激活任务

↓

同步主材

↓

同步考核

↓

同步排程

↓

通知OMS
```

注意：

OMS：

只有：

```
紧急立即
```

才调用。

这一点建议单独强调。

---

# 第六章：为什么要这样设计（设计思想）

这是很多技术方案都缺的一章。

实际上你们方案最大的价值就在这里。

建议增加。

例如：

### 为什么先落表？

避免：

```
A→B→C→D

连续修改四次。
```

---

### 为什么T+1？

统一聚合：

```
最终：

A→D
```

减少：

任务创建。

---

### 为什么要经过需求层？

因为：

需求：

才是真实业务。

任务：

只是需求产生的结果。

所以：

应该：

```
需求

↓

任务
```

而不是：

```
任务

↓

需求
```

---

### 为什么需求层再回调？

为了：

保证：

所有系统：

都监听：

同一个事件。

而不是：

每个系统：

自己处理。
