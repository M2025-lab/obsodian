# 1 背景

排程系统以品类+供应商为维度管理材料需求，每个 SKU 绑定一个供应商。供应商汰换是高频业务场景（供应商退出、产能不足、质量问题等），但系统缺乏"信息版本化"机制——汰换后旧版本信息如何处置、新版本如何生成、两个版本如何衔接，没有明确规则。

本次改造分三层：需求层、排程计划、考核。本文档聚焦**需求层**，负责根据汰换类型和货单/任务状态判定履约承接方（老商继续 or 新商跟进），供调度层调用。

# 2 名词解释

- **汰换类型**：商品中心下发的本次汰换性质，三选一：
    
    - 紧急汰换-立即切换
        
    - 紧急汰换-不立即切换
        
    - 常规汰换
        
- **货单状态**：履约区段下单后的货单生命周期阶段。紧急汰换用三档（无货单 / 待接单·备货中 / 待发货·待签收），常规汰换只用二档（有货单 / 无货单）
    
- **履约承接方**：本次汰换后由谁继续履约——老商继续 or 新商跟进
    
- **货前服务处理人角色+完成状态**：常规汰换判断二的依据——是否存在「处理人为供应商角色 且 ≥1 个已完成」的货前任务
    
- **调度层**：监听商品中心汰换事件、捞取影响范围（受影响的 SKU/项目）、调用需求层判定接口、驱动排程和考核的协调层
    

# 3 需求理解

## 3.1 需求层定位

需求层的职责是**为调度层提供履约承接判定能力**。调度层监听商品中心汰换事件并捞取影响范围后，调用需求层接口，需求层查询货单/任务状态并判定承接方，结论返回给调度层。

**只做判定，不做执行**：需求层只回答"谁来履约"，不处理任务重排、取消、考核重算（这些由调度层和排程/考核侧各自处理）。

## 3.2 调用链路

```Plain
商品中心
  │
  │ Kafka
  ▼
调度层
  │  · 监听汰换事件
  │  · 捞取影响范围（受影响的 SKU、项目订单）
  │
  │ RPC/HTTP: 调用需求层判定接口
  ▼
需求层（本文档范围）
  · 查询货单状态（SCM-Order）
  · 查询货前任务完成情况（排程）
  · 执行承接判定逻辑
  · 返回判定结论给调度层
```

## 3.3 输入（调度层传给需求层）

- SKU ID + 商户 ID
    
- 品类 ID
    
- 汰换前供应商（code/名称）
    
- 汰换后供应商（code/名称）
    
- 汰换类型：紧急汰换-立即切换 / 紧急汰换-不立即切换 / 常规汰换
    
- 项目订单 ID（用于查货单和任务）
    

## 3.4 核心逻辑

按汰换类型分流：

- **紧急汰换**（立即/不立即）→ 查货单状态，按矩阵判定
    
- **常规汰换** → 两步判断
    

判定结论只有两种：**老商继续** or **新商跟进**，返回给调度层。

## 3.5 RD 引申需求

- 汰换是一次性切换动作，不是持续生效的状态机
    
- 判定逻辑仅依赖汰换事件发生那一刻的货单状态和任务完成情况
    
- 接口幂等：同一事件 + 同一 SKU 返回已有结果
    
- 判定结论持久化到 receipt_order_service_item 表，供追溯
    

# 4 设计目标

1. 提供 RPC/HTTP 接口供调度层调用，入参为单个 SKU 的汰换信息
    
2. 实现履约承接判定逻辑（紧急汰换矩阵 + 常规汰换两步判断）
    
3. 判定结论持久化到 receipt_order_service_item 表
    
4. 接口幂等：同一 eventId + skuId + projectOrderId 不重复判定

## 5.3 数据结构

### 5.3.1 扩展表：receipt_order_service_item

在现有 `receipt_order_service_item` 表增加字段，记录汰换信息和判定结果：

- supplier_code_before (VARCHAR 64)：汰换前供应商code，NULL=未发生汰换
    
- supplier_name_before (VARCHAR 128)：汰换前供应商名称
    
- replacement_type (TINYINT)：汰换类型，1-紧急立即 2-紧急不立即 3-常规，NULL=未汰换
    
- replacement_event_id (VARCHAR 64)：汰换事件ID，幂等键，(replacement_event_id + sku + project_order_id) 联合唯一索引
    
- fulfillment_decision (TINYINT)：履约承接方，1-老商继续 2-新商跟进
    
- decision_reason (VARCHAR 256)：判定理由
    
- decision_time (DATETIME)：判定时间
    

**设计说明**：

- 原 `supplier_code` / `supplier_name` 字段汰换后更新为新供应商信息
    
- 旧供应商信息存入 `supplier_code_before` / `supplier_name_before`
    
- `replacement_event_id` 为空表示此行未发生汰换，保持原有行为不变
    
- 幂等由 `replacement_event_id + sku_id + project_order_id` 组合保证（同一次事件同个 SKU 同个项目不重复判定）

## 5.5 特殊设计：履约承接判定逻辑

### 5.5.1 紧急汰换判定矩阵

输入：货单状态（三档）+ 汰换类型（立即/不立即）输出：老商继续 / 新商跟进

```Plain
│ 货单状态 ╲ 汰换类型    │ 紧急-立即切换 │ 紧急-不立即切换 │
├────────────────────────┼────────────┼────────────────┤
│ 无货单（货单未生成）     │ 新商跟进     │ 新商跟进         │
│ 有货单·待接单/备货中    │ 新商跟进     │ 老商继续         │
│ 有货单·待发货/待签收    │ 老商继续     │ 老商继续         │
```

**一句话口径**：无货单 → 一律新商跟进；有货单 → 仅「紧急-立即 + 待接单/备货中」新商跟进，其余全部老商继续。

### 5.5.2 常规汰换两步判断

**判断一：是否有货单？**

- 有货单（无论生命周期阶段）→ 货单记录的供应商继续履约（老商继续），结束
    
- 无货单 → 进入判断二
    

**判断二：货前服务中是否存在「处理人为供应商角色 且 ≥1 个已完成」的任务？**

- 存在 → 该供应商继续履约（老商继续），结束
    
- 不存在 → 新商跟进
    

**设计理由**：

- 判断一：货单已生成意味着履约事实已落在该供应商，由其继续完成更顺，避免货单迁移成本
    
- 判断二：无货单但货前服务已有该供应商角色的完成项，说明该供应商已实质投入；否则说明尚未实质投入，可以切给新商

# 6 风险评估

- **调度层接口约定未定**：请求/响应格式需与调度层对齐
    
- **货单状态查询接口**：需确认 SCM-Order 是否支持按 projectOrderId + skuId 查货单状态
    
- **货前任务查询接口**：需确认排程侧是否能按 projectOrderId + skuId + categoryId 查询，以及是否能区分处理人角色
    
- **receipt_order_service_item 行级匹配**：需确认调度层传入的 projectOrderId + skuId 能唯一定位到 receipt_order_service_item 中的一行，否则需要调度层传 itemId

# 8 自测case

1. 紧急-立即切换 + 无货单 → 判定新商跟进
    
2. 紧急-立即切换 + 待接单/备货中 → 判定新商跟进
    
3. 紧急-立即切换 + 待发货/待签收 → 判定老商继续
    
4. 紧急-不立即切换 + 无货单 → 判定新商跟进
    
5. 紧急-不立即切换 + 有货单（任意状态）→ 判定老商继续
    
6. 常规汰换 + 有货单 → 判定老商继续（不进入判断二）
    
7. 常规汰换 + 无货单 + 货前有供应商角色已完成任务 → 判定老商继续
    
8. 常规汰换 + 无货单 + 货前无供应商角色或无已完成任务 → 判定新商跟进
    
9. 重复 eventId + skuId + projectOrderId → 幂等，返回已有判定结果
    
10. receipt_order_service_item 记录正确更新（原 supplier 移到 before，新 supplier 更新到主字段）
