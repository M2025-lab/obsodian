## 一、系统知识（System Knowledge）

### 1.1 项目定位
edar-starlord 是链家/贝壳体系下**材料域**的核心业务系统，负责**主材任务（task_dispatch）的全生命周期管理**——覆盖主材的复尺（测量）、报价、排程、派发、施工履约、供应商协同等环节。系统以"项目订单（projectOrderId）+ SKU + 供应商/商户（merchantId）"为主维度，编排主材任务的创建、激活、履约与完结。

### 1.2 技术栈与模块划分
- **语言/框架**：Java + Spring（MVC）+ MyBatis；多模块 Maven 工程。
- **包路径**：`com.ke.utopia`。
- **分层模块**：

| 模块 | 职责 |
|------|------|
| `edar-starlord-api` | DTO / Bean / 枚举（如 `SupplierReplaceHiveDTO`、`SupplierReplaceMessageCondition`） |
| `edar-starlord-base` | 基础能力 |
| `edar-starlord-dao` | MyBatis Mapper（337 个 xml，核心表 `task_dispatch` / `task_dispatch_node` / `measure_apply` / `supplier_replace_message`） |
| `edar-starlord-manager` | 业务编排 / 外部调用（含 Hive 大数据调用、企业微信通知等） |
| `edar-starlord-service` | 核心业务逻辑、事件监听、供应商汰换编排 |
| `edar-starlord-web` | Controller、后门（BackDoor）、静态页 |

### 1.3 核心领域模型
- **主材任务 `task_dispatch`**：任务派发主表，含 `process_status`(流程状态)、`supplier_code/supplier_name`(供应商)、`task_type`(任务类型)、`material_code/name`(材料)、`order_no`(关联货单)、`source_type`、`service_mode`、`estimated_time` 等。
- **任务节点 `task_dispatch_node`**：任务的工序节点（`node_type`、操作人 `operator_id`）。
- **复尺/测量申请 `measure_apply`**：主材测量的业务单，是供应商汰换的数据源头之一。
- **供应商汰换消息 `supplier_replace_message`**：记录"某 SKU + 商户"维度的汰换处理消息，含 `status`(处理中/完成) 与 `fulfill_result`(履约结果：老商/新商)。

### 1.4 主材任务生命周期（主线）
复尺申请 → 生成主材任务（`createTask`）→ 任务打标（`extend.replacement`）→ 激活（到激活时间）→ 并行执行（货主材+考核 / 排程 / 下单）→ 履约（老商或新商）→ OMS 货单处理 → 完结。

## 二、业务逻辑（Business Logic）：供应商汰换

### 2.1 业务背景
当主材的供应商需要被"汰换"（替换）时——即某个 SKU 在某商户下由**老供应商**切换为**新供应商**——系统需要把老商相关的任务、货单平滑迁移到新商，并保证复尺单、排程、下单、履约（OMS）等环节正确衔接，避免重复创建或遗漏。

### 2.2 触发入口
需求层通过消息 `measure-apply-order-supplier-replacement` 下发汰换指令，由事件处理器
`SupplierReplaceDemandEventHandler` 消费，调用 `SupplierReplaceProcessService.process()` 进行编排。

### 2.3 编排流程（3.1 ~ 3.6）
`SupplierReplaceProcessServiceImpl.process()` 按如下步骤处理每个 SKU 变更明细（`SkuChangeDetail`，含 `skuId`/`merchantId`/`newSupplierCode` 等）：

| 步骤 | 说明 |
|------|------|
| **3.1 老商履约** | 老商已履约的，直接走主材申请单逻辑结束该 SKU 处理 |
| **3.2 生成新商任务** | 调 `ScmMeasureApplyService.createTask` 生成新商主材任务+下单任务，查回新商任务并对 `extend.replacement` 打标 |
| **3.3 激活** | 激活时间 < 当前时间则立即激活任务 |
| **3.4 状态复制** | 新商履约 + 新老商均非供应商（a 场景）时，将老商任务状态/节点复制到新商（copyTaskState / copyNodeState） |
| **3.5 并行后置** | 并行执行：货主材+考核、排程（demandChange）、下单 |
| **3.6 紧急汰换→OMS** | `replace_type=2`（紧急+立即汰换）时，先经 **Hive 大数据反查已发货货单**，再调 OMS 取消并重下新商货单 |

> 关键闭环：第 **3.6 步**依赖 Hive 大数据查询能力——通过 `SupplierReplaceHiveManager.queryOrderRange(skuId, merchantId)` 反查"项目单/订单号/订单状态/供应商编码"，命中已发货货单后才驱动 OMS 取消重下。这正是本需求中大数据能力接入的意义所在。

### 2.4 结果标记
全部处理完成后，将"处理中（PROCESSING）"的供应商汰换消息（`supplier_replace_message`）按 `skuId + merchantId` 维度标记为"已完成（fulfill_result = 新商履约 = 1）"，实现需求层回调闭环。

### 3.1 HiveAPI 大数据拉取能力接入（核心贡献）
**提交**：`76c73b73b [紧急]HiveAPI拉取数据功能`（2026-07-14）+ 多笔测试提交
**背景**：供应商汰换第 3.6 步需要从大数据平台反查已发货货单，但系统原本缺乏可复用的 Hive 查询通道。

**新增/改动内容**：
1. **`HiveApiNewUtil`（Spring 组件）**：`edar-starlord-manager/.../util/hive/HiveApiNewUtil.java`
   - 以 `@Component` 形式托管，从配置注入 `key`/`secret`；
   - 提供简洁入口 `searchHiveApi(Map<String,String> param, String url)`，内部委托既有 `HiveApiUtil.searchHiveApi`（含签名、OkHttp 调用、结果解析）。相比旧版静态工具类，更契合 Spring 注入式调用。
2. **`SupplierReplaceHiveManager` 接口 + `Impl`**：
   - 常量 `HIVE_URL = http://i.data.api.lianjia.com/v2/meta/queryProjectSkuidOrderNo`；
   - 方法 `queryOrderRange(List<SkuMerchantPair> pairs)`：按 `skuId + merchantId` 批量反查 `projectOrderId / orderNo / orderStatus / skuId / merchantId / supplierCode`，返回 `List<SupplierReplaceHiveDTO>`。
3. **`SupplierReplaceHiveDTO`**：`edar-starlord-api/.../dto/SupplierReplaceHiveDTO.java`（可序列化传输对象）。
4. **后门入口**：`BackDoorController` 新增 `POST /backdoor/hiveApi/search`
   - 入参 `@RequestBody Map<String,String> param` + `@RequestParam String url`；
   - 自动注入分区参数 `pt = 昨天(yyyyMMdd)000000`，便于联调大数据。
5. **POM 依赖**：`edar-starlord-manager/pom.xml` 新增 `okhttp`、`logging-interceptor`、`guava-retrying`（2.0.0，重试支持）。
6. **测试**：`SupplierReplaceApiTest`（后门/本地联调用）。

**业务价值**：打通了主材系统与大数据平台的实时查询通道，使供应商汰换第 3.6 步能精准识别已发货货单并触发 OMS 取消重下，避免错发/漏发。

### 3.2 需求层回调新增"汰换处理成功标记"
**提交**：`d33479452 [紧急]供应商汰换需求，需求层回调新增汰换处理成功标记`（2026-07-21）

**改动**：在 `SupplierReplaceProcessServiceImpl.process()` 中新增第 5 步 `markReplaceMessagesDone(bo)`：
- 注入 `SupplierReplaceMessageService` 与 `SupplierReplaceMessageDao`；
- 收集 `notChange / add / cancel / modify` 四个 SKU 变更列表中的 `skuId_merchantId` 维度；
- 反查 `PROCESSING`(处理中) 状态的 `supplier_replace_message`，逐条 `markDone(..., fulfillResult=新商履约=1)`；
- 新增私有方法 `markReplaceMessagesDone` / `collectSkuMerchantPairs` / `collectMeasureDetailPairs`，并对每层做空值保护。

**业务价值**：补全了需求层回调的闭环——汰换处理成功后显式标记消息完成，下游可感知履约结果，避免消息长期停留在"处理中"导致重复处理。

### 3.3 新老商任务处理空指针修复
**提交**：`dcd1e8409 [紧急]新老商任务处理`（2026-07-20）

**改动**：步骤 3.4 原代码：
```java
if (this.isAllNotSupplierConfig(oldTask, newTask, skuChangeDetail.getNewSupplierCode())) {
```
修复为：
```java
if (newTask != null && this.isAllNotSupplierConfig(oldTask, newTask, skuChangeDetail.getNewSupplierCode())) {
```
**原因**：当老商任务**不存在对应新商任务**（`newTask == null`）时，原逻辑直接调用 `isAllNotSupplierConfig` 触发 **NPE**，导致整批汰换处理中断。增加 `newTask != null` 判空后，无新商任务的 SKU 被安全跳过，流程健壮。
