01 业务认知

开发视角：材料履约要"管什么"

材料履约本质是协调"货前测量 → 货后安装验收"的全过程，starlord 处在家装链路最末端的交付层。落到职责上，它要管好 4 件事 —— 也是后面几页的地图。

① 管主材任务全生命周期

  • 把"客户要哪些主材"拆成可追踪任务流
  • 以 下单 为货前/货后唯一分界

② 管履约流程配置

  • 按 分公司 / 品类 / 套餐 适配流程（非固定）
  • 可配执行角色、重启、耗时、上门时间 → 复杂度根源（见系统认知）

③ 管发货与验收调度

  • 发货规则：单品一次发、多品集中发，按工地进度约发货日
  • 验收专人跟进、多工地需调控验收时间

④ 管现场执行衔接

  • 施工包（人+料）是安装单位
  • 用工平台协调人/工地/工具/材料/工序/工艺

这 4 件事里，① 主材任务生命周期是主线（下一页看它的一生）；② 流程配置是系统复杂度根源（系统认知深入）；③④ 现场衔接靠施工包与用工平台落地（再下一页）。串起这一切的核心载体，就是 主材任务。

01 业务认知

全链路靠谁落地：两类协作实体

全链路要真正跑起来，靠两类实体协作：施工包是"安装"的执行单位，用工平台是"人、货、工序"的协调中枢。两者把抽象的 SKU 履约流程变成具体的现场作业。

施工包

工人施工动作的管理载体，以「人 + 料」构成，安装以施工包为单位；支持基装/安装、整装/零售、总工/产业/供应商工人等多种模式，涉及生成、变更、延期、验收、拆包/合包。

用工平台

协调「什么人在什么工地、用什么工具、对什么材料、按工序、施加工艺」，依据配置与工期协调人货进场与安装，是履约交付的重点。

以前置系统生成的主材申请单为业务起点，将需求拆分为 SKU；以下单作为货前/货后分界，用货单号串联供应商履约全过程，并通过商品变更消息、数据聚合、订单关联机制，保证供应商、SKU 与履约任务的一致性。

配置不是静态的：启动时把 processChain（JSON 有向图，节点=状态/步骤、边=流转） 加载进内存，由它驱动任务的创建与状态流转。配置模式(mode_type) → 模板 → 流程定义 → 状态机，是一条可验证的因果链。

模板如何决定任务流程

每个主材按 品类 × 城市 × 套餐/服务模式 × 配置模式 匹配模板；模板内含 processChain 决定该主材经历的节点序列——不同模板 = 不同流程：

测量→下单→送货→安装标准
·
测量→变更单→下单→送货含变更
·
测量→设计→复尺→…→安装→验收整装

生命周期

创建后 NOT_ACTIVE → 到期/条件满足激活进入 UNCOMPLETED → 逐节点流转 → 全部完成进入 COMPLETED；前置不满足则挂起 SUSPEND_ACTIVE。

激活的坑

受前置条件约束（尾款拦截 / 订单状态 / 依赖节点）；存在新版 / 旧版双激活路径，新版还需处理多分支、条件路由。

02 系统与技术认知

状态机落到两张表：主材任务 vs 主材任务节点

抽象的状态机由两张表承载，对应一条层级链：项目 → 多个主材 → 每主材按模板生成多个主材任务（task_dispatch，如「测量」一条）→ 每任务内部多个工序节点（task_dispatch_node）。node_task 保存有序节点序列，current_node_type 随完成向前推进，即"流转到哪个流程"。

主材任务 · task_dispatch

颗粒度 = 一次履约，task_type（1测量/3复尺/4下单/5送货/6安装/7接单/8备货/10设计/12排产/14进仓/200变更单…）。状态 4 态（含 4 暂停激活）。含 plan_activate_time、supplier_code、order_no 等。

任务节点 · task_dispatch_node

颗粒度 = 一道工序，node_type 阶梯：0默认→20通知可启动→40通知启动→60启动→65自检验收→80启动完成→85业主确认。状态 3 态（无暂停）。表中还记录激活时间(plan_activate_time/estimated_time/start_time/current_node_time) 与处理人(executor_type/executor_id/submit_by) 等。

「激活」的两个层级 —— 复杂度与并发坑的根源
维度任务激活（task 级）节点激活（node 级）
颗粒度一次 SKU 履约一道工序

状态数4 态（含 暂停激活）3 态（无暂停）

三类复杂点 · 详细拆解（下一阶段的主攻方向）

把前几页串起来：配置维度爆炸(第8页) + 状态机与领域模型(第9/9.5页) + 跨系统一致性(上下游/下一页) 三者叠加，构成系统真正的复杂度。归纳如下，也是我下一阶段要逐个拆解的目标。
复杂点 ①
配置维度爆炸
  • 三套配置并存：mode5/6/7 走完全不同的配置链路
  • 规则单元多维：单据类型 × 业务类型 × 分公司 × 套餐
  • 节点属性多表：n_material_template / node_cfg / route / task_cfg / time_cfg / push_cfg 等相互关联
  • → 配置易错、排查难、上手成本高
怎么拆：理清 5/6/7 映射 → 梳理核心配置表字段与关联键 → 形成配置全景图 + 校验清单
复杂点 ②
状态机与激活复杂
  • 节点类型多：1/20/40/50/60/65/80/85，随任务类型不同而不同
  • 双激活路径：新版（流程定义 + 多分支路由）vs 旧版（激活首节点）
  • 前置拦截：尾款 / 订单状态 / 节点依赖 / 挂起重启 / 驳回
  • 竞争场景：创建异步激活 + 回调同步激活两次调用（已在汰换中验证 CAS+事务保护正确性）
怎么拆：梳理每类任务的触发路径 / 状态流转 / 接口出入参，画全状态机，找出异常分支
复杂点 ③
跨系统一致性
  • 联动系统多：需求层 / OMS / SDM / VSS / Cube / 排程 / 考核
  • 消息链路长：MQ 落库 → T+1 聚合 → 需求层回调 → 任务操作 → 下游同步 → OMS 换单
  • 典型问题：状态未闭环（本次已修 markDone）、A→B→C→D 需聚合去重、两次激活竞争
怎么拆：画消息时序图标注每个状态转换点，梳理巡检补偿（RefreshDataService / 事件重试）兜底

方案①：核心业务规则 · 谁来履约

方案的第一步是判断"谁来履约"。规则围绕货单状态与汰换类型展开 —— 核心原则：已发货的货单不轻易动，避免错发漏发。
货单状态 ╲ 汰换类型紧急-立即紧急-不立即
无货单新商跟进新商跟进
有货单·待接单/备货中新商跟进老商继续
有货单·待发货/待签收老商继续老商继续

紧急汰换判定矩阵：无货单一律新商跟进；有货单仅「紧急-立即 + 待接单/备货中」切新商，其余留老商。

常规汰换 · 两步判断

  • 判断一：有货单 → 老商继续；无 → 进入判断二
  • 判断二：货前有「供应商角色且 ≥1 已完成」任务 → 老商继续；否则新商跟进

角色场景 a / b

  • a 场景：新老商均非供应商 → 老商状态全量覆写新商（保连续性）
  • b 场景：含供应商角色 → 老商在途则取消老商，按实际生成新商
  • 角色任一不可判断 → 默认 b

  • supplier_replace_message 汰换消息：status / fulfill_result

主材任务生命周期：复尺申请 → createTask → 打标 extend.replacement → 激活 → 并行（货主材+考核 / 排程 / 下单）→ 履约（老/新商）→ OMS 货单 → 完结。

03 我的实践产出

方案④：设计思想 & 数据设计要点

为什么是上面这套方案（落表→T+1→需求层→回调），而不是即时改任务？每一处取舍都为了"一致性"与"减少无效任务"。

为什么这样设计

  • 先落表：避免 A→B→C→D 连续改四次任务
  • T+1 聚合：收敛为最终态 A→D，减少任务创建、消除无效任务
  • 经需求层：需求才是真实业务，任务由需求产生 → 需求↓任务
  • 需求层再回调：各外围系统监听同一事件，从源头保证一致性

数据设计

  • 消息落库：MerchantSkuSupplierReplacePayload → supplier_replace_message
  • 中间表：status 0/1/2/3，idx_status_create 聚合捞取
  • 需求层接口：POST /api/supplier-replacement/evaluate 只判「谁来履约」，结论落 receipt_order_service_item
  • Hive 取数：HiveApiUtil.searchHiveApi 拉受影响订单，分批 1000

跨团队协作：需求横跨 商品中心 / 需求层 / OMS / 排程 / 考核 / Hive 多团队；我聚焦在 回调层状态闭环、激活并发、Hive 取数链路 几个接合点。
