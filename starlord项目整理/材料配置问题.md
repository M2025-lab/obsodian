结合这张「节点配置编辑页」，我把界面上**每一个可配置项**逐一对应到数据库表和字段，你可以顺着“界面操作→数据落在哪张表”的思路快速建立映射，不用死记硬背表结构。

### 整体定位
这个页面属于 **`n_` 前缀的新一代主流程配置体系**，编辑的是「浴室柜台面」这套流程里的**安装节点**的详细规则。所有配置最终会落到 4 类表里：
1. 节点基础属性 → `n_material_node_cfg`
2. 节点业务规则（角色、重启逻辑）→ `n_material_task_cfg`
3. 各类考核时间规则 → `n_material_time_cfg` + `n_material_time_relation`
4. 节点间流转连线 → `n_material_route`
5. 生效范围（分公司/套餐/供应商）→ `n_material_template_unit`

## 一、右侧面板：每个配置项对应哪张表
### 1. 节点类型：安装
- **对应表**：`n_material_node_cfg`（主材节点定义表）
- **对应字段**：`node_type`（节点类型枚举，存数字编码，对应“安装”）
- **存的是什么**：定义这个节点的身份类型，是节点的基础标识，全流程靠这个字段区分“通知安装、安装、安装验收”等不同环节。

### 2. 平台考核时间：基装验收 之后 11 工作日
- **对应表**：`n_material_time_cfg`（考核时间配置表），`time_type = 1`（平台考核）
- **字段拆解**：
  - 依赖基准节点：关联 `n_material_time_relation` 表，`relation_code` 对应“基装验收”节点
  - 时间相对关系：`schedule_relation = 3`（1=之前，2=当天，3=之后）
  - 间隔时长：`duration = 11`
  - 天数类型：`duration_type = 2`（1=自然日，2=工作日）
- **存的是什么**：平台对这个节点的整体考核时效。意思是：基装验收完成后，11个工作日内必须完成安装，否则算平台侧超期。

### 3. 首次执行角色：供应商
- **对应表**：`n_material_task_cfg`（主材节点业务配置表）
- **对应字段**：`role_types`（多个角色逗号分隔，存供应商对应的角色编码）
- **存的是什么**：节点第一次激活时，由哪个角色负责处理操作。这里安装节点的默认执行人是供应商。

### 4. 首次角色考核时间：通知安装_可上... 之后 3 工作日
- **对应表**：`n_material_time_cfg`，`time_type = 2`（首次角色考核）
- **字段拆解**：
  - 依赖基准节点：`relation_code` 对应“通知安装-可上门”节点
  - 时间关系：`schedule_relation = 3`（之后）
  - 间隔天数：`duration = 3`
  - 天数类型：`duration_type = 2`（工作日）
- **存的是什么**：对执行角色（供应商）的首次考核时效。通知安装发出后，3个工作日内供应商必须完成安装作业，是对执行方的时效考核。
- 右侧「添加时间」按钮：可以给同一个节点加多条考核规则，对应表中同节点、同 `time_type` 下的多条记录。

### 5. 不合格是否重启：重启
- **对应表**：`n_material_task_cfg`
- **对应字段**：通过 `restart_node_type` 是否为 0 判断，非 0 代表启用重启
- **存的是什么**：节点验收不合格时，是否触发回流重启流程。

### 6. 重启节点/任务：通知安装
- **对应表**：`n_material_task_cfg`
- **对应字段**：`restart_node_type`（存“通知安装”对应的节点类型编码）
- **存的是什么**：不合格重启后，流程回到哪个节点重新执行。这里安装验收不合格，就退回到「通知安装」节点，重新派单安排安装。
- 补充：这条回流逻辑同时会在 `n_material_route`（流程路由表）生成一条条件连线：源节点=安装验收，目标节点=通知安装，规则表达式=“不合格”。

### 7. 重启执行角色：供应商
- **对应表**：`n_material_task_cfg`
- **对应字段**：`restart_role_types`
- **存的是什么**：重启后的节点由哪个角色执行，这里重启后依然由供应商负责。

### 8. 重启角色考核时间：通知安装_可上... 之后 3 工作日
- **对应表**：`n_material_time_cfg`，`time_type = 3`（重启考核）
- **字段拆解**：和首次考核结构完全一致，只是 `time_type` 不同
- **存的是什么**：节点重启后，重新计算的考核时效，规则独立于首次考核。

## 二、左侧流程图对应的数据表
左边的「开始 → 通知安装 → 安装 → 安装验收 → 结束」流程图，对应两类数据：
1. **每个绿色节点**：对应 `n_material_node_cfg` 里的一条记录，每个节点有独立的 `node_code`、`node_type`。
2. **实线箭头（正常流转）**：对应 `n_material_route` 里的正常路由
   - `source_code` = 前置节点编码
   - `target_code` = 后置节点编码
   - 无条件，默认完成就走这条线
3. **虚线“重”箭头（重启回流）**：对应 `n_material_route` 里的条件路由
   - `source_code` = 安装验收
   - `target_code` = 通知安装/安装
   - `rule_expression` = 不合格条件表达式

## 三、顶部生效范围对应的数据表
顶部的「不限制供应商、应用分公司、应用套餐」这些维度信息：
- **对应表**：`n_material_template_unit`（用工任务模板维度表）
- **对应字段**：
  - `material_code` / `material_name`：主材（浴室柜台面）
  - `supplier_code`：供应商（不限制则为通配）
  - `mdm_code`：分公司
  - `product_combo_id`：产品套餐
- **存的是什么**：这套流程规则在哪些业务场景下生效，是“项目下单时匹配哪套流程”的判断依据。

## 四、快速掌握所有表的通用方法
以后再看到任何配置页面，按这个思路对应：
1. **选“哪个节点/哪套流程”** → 找 `n_material_node_cfg` / `n_material_process_define`
2. **“谁来做、能不能重启、怎么合并”等业务规则** → 找 `n_material_task_cfg`
3. **“多久完成、从哪天算、工作日/自然日”等时间规则** → 找 `n_material_time_cfg`
4. **“节点之间怎么走、什么条件走哪条线”** → 找 `n_material_route`
5. **“哪些分公司/套餐/主材能用”** → 找维度绑定表 `n_material_template_unit`
