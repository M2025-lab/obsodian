## 一、方法签名

```java
public void activateTaskDispatch(TaskDispatch taskDispatch, MaterialTask materialTask)
```

### 调用来源

被 `activateTaskDispatch(TaskDispatch)`（重载方法，第 216–231 行）调用，条件为：

| 前置判断 | 说明 |
|----------|------|
| `processStatus == UN_ACTIVE(1)` | 只有未激活的任务才需要激活 |
| `planActivateTime` 在 [2000-01-01, 今天结束) 范围内 | 只有"计划在今天激活"的任务才执行 |
| `processCode` 为空 | 有 processCode 的走 v2 激活（`materialActivateV2Service`），不走此方法 |

## 二、方法整体流程

```
activateTaskDispatch(taskDispatch, materialTask)
  │
  ├─ ① processCode 二次检查 → v2 激活?
  │     └─ 非空 → materialActivateV2Service, return
  │
  ├─ ② shouldActivate() 检查?
  │     ├─ PLAN_TIME: 查订单状态是否 ≥ 配置工序状态
```

#### `shouldActivate` 方法（第 286–310 行）

```java
private boolean shouldActivate(TaskDispatch taskDispatch, MaterialTask materialTask) {
    if (Objects.isNull(taskDispatch) || Objects.isNull(materialTask)) {
        return true;  // 参数为空，默认允许激活
    }

    ScheduleRelationEnum relationEnum = ScheduleRelationEnum.getByValue(
        Integer.valueOf(materialTask.getActivateScheduleRelation()));

    // 分支 A：PLAN_TIME 模式 + 关系是 TODAY 或 AFTER
    if (ActivateModeEnum.isPlanTime(materialTask.getActivateMode().intValue())
        && afterTodayRelationList.contains(relationEnum)) {

        ProjectOrderDetailBO projectOrder = projectOrderManager.getProjectOrder(
            taskDispatch.getProjectOrderId());
        return MajorSequenceEnum.canActivateByProjectOrderStatus(
            projectOrder.getStatus(), materialTask.getActivateRelationCode());
    }

    // 分支 B：DEPENDENT_NODE 模式
    if (ActivateModeEnum.isDependentNode(materialTask.getActivateMode().intValue())) {
        List<TaskDispatch> taskDispatchList = this.queryDependentNodeTask(taskDispatch);
        List<TaskDispatch> unFinishedList = taskDispatchList.stream()
            .filter(e -> ProcessStatusEnum.FINISHED.getValue().byteValue() != e.getProcessStatus())
            .collect(Collectors.toList());
        return CollectionUtils.isEmpty(unFinishedList);  // 全部完成才返回true
    }

    return true;  // 其他模式默认允许激活
}
```

#### 分支 A：PLAN_TIME 模式的判断

| 条件 | 判断依据 | 数据来源 |
|------|---------|---------|
| `activateMode == 0` | `material_task.activate_mode` | 配置表字段 |
| `activateScheduleRelation` 是 `TODAY(2)` 或 `AFTER(3)` | `material_task.activate_schedule_relation` | 配置表字段 |
| 订单状态是否 ≥ 配置工序对应的状态 | `projectOrder.status` vs `material_task.activate_relation_code` | 外部 Feign + 配置表 |
| 订单是否已取消 | `MajorSequenceEnum.canActivateByProjectOrderStatus` 内部判断 | 枚举逻辑 |

**查的配置**：
- `material_task.activate_mode` — 激活模式
- `material_task.activate_schedule_relation` — 时间关系
- `material_task.activate_relation_code` — 关联工序 code

**查的外部服务**：
- `projectOrderManager.getProjectOrder(projectOrderId)` → 获取 `ProjectOrderDetailBO`（包含订单状态）

**判断本质**：
```
订单施工进度 是否已经推进到了 模板配置的工序节点？
例如：激活条件为"瓦工完工后3天" → 判断订单状态是否已经≥"瓦工完工"
```

#### 分支 B：DEPENDENT_NODE 模式的判断

| 条件 | 判断依据 | 数据来源 |
|------|---------|---------|
| `activateMode == 2` | `material_task.activate_mode` | 配置表字段 |
| 是否有前置任务未完成 | 查路由表 + 查实例表 | 见下方 |

**查的配置**（`queryDependentNodeTask`，第 760–782 行）：

```sql
-- 1. 查路由表：找出当前任务依赖哪些前置模板
SELECT source_id FROM material_task_route_instance
WHERE target_id = #{taskDispatch.templateId}
  AND state = 1

-- 2. 查实例表：找出这些前置模板对应的实例
SELECT * FROM task_dispatch
WHERE template_id IN (source_ids)
  AND project_order_id = #{taskDispatch.projectOrderId}
  AND material_code = #{taskDispatch.materialCode}
  AND supplier_code = #{taskDispatch.supplierCode}
  AND order_no = #{taskDispatch.orderNo}
```

**判断逻辑**：

| 场景 | 结果 |
|------|------|
| 无路由记录（无前置依赖） | `return true`（无依赖 = 可以激活） |
| 有前置任务，但存在状态的不是 `FINISHED(3)` 的 | `return false`（有未完成的依赖 → 不能激活） |
| 有前置任务，全部是 `FINISHED(3)` | `return true`（依赖全部完成 → 可以激活） |

#### 被拦截时的处理（`updateTaskDispatch`，第 869–875 行）

```java
private void updateTaskDispatch(TaskDispatch taskDispatch) {
    TaskDispatch updateTaskDispatch = new TaskDispatch();
    updateTaskDispatch.setId(taskDispatch.getId());
    updateTaskDispatch.setProcessStatus(
        TaskDispatchStatusEnum.SUSPEND_ACTIVE.getValue().byteValue());  // 设为 4
    taskDispatchDao.updateById(updateTaskDispatch);
}
```

**状态变化**：`UN_ACTIVE(1)` → `SUSPEND_ACTIVE(4)`

> `SUSPEND_ACTIVE` 表示"条件未满足，暂时挂起"，与 `UN_ACTIVE` 的区别是：
> - `UN_ACTIVE` — 纯粹没到激活时间，定时任务会继续尝试
> - `SUSPEND_ACTIVE` — 被特定条件拦截（尾款），需要外部事件（付款成功）触发重新激活

### 步骤 ⑦ — 激活后置处理（第 273–278 行）

#### 7.1 `completeAssignerTaskWhenActivate`（第 313–323 行）

```java
public void completeAssignerTaskWhenActivate(TaskDispatchNode node, TaskDispatch taskDispatch) {
    String packageCode = fetchPackageCode(taskDispatch.getId());  // 查关联的施工包
    if (StringUtils.isNotBlank(packageCode)) {
        completeTaskNode(taskDispatch, node, packageCode);        // 有施工包 → 派单员任务自动完成
    } else {
        installerTaskService.completeAssignerTask(node, taskDispatch);  // 无施工包 → 原方式处理
    }
}
```

**`fetchPackageCode`**（第 325–332 行）：

```java
private String fetchPackageCode(Long taskDispatchId) {
    List<TaskDispatchNode> nodes = taskDispatchNodeDao.listByTaskId(taskDispatchId);
    // 找第一个有 packageCode 的节点
    Optional<TaskDispatchNode> tempNode = nodes.stream()
        .filter((node) -> StringUtils.isNotBlank(node.getPackageCode())).findFirst();
    return tempNode.isPresent() ? tempNode.get().getPackageCode() : "";
}
```

**`completeTaskNode`**（第 335–363 行）— 有施工包时的处理：

```
判断: 
  ① taskType == INSTALL(6) 或 ONCE_INSTALL(9)?
  ② nodeType == NOTIFY_CAN_START(20) 或 NOTIFY_START(40)?
  两者都满足才继续，否则 return

查施工包状态（packageConstructionManager.queryBaseInfoByPackageCode）:
  施工包状态为 WAIT_APPROACH / PROCESSING / REJECT_PROCESSING / ACCEPTING / COMPLETE?
    → 自动完成"安装工派单"节点（taskDispatchService.completeInstallerAssignerNode）
  其他状态 → 不做处理
```

> **业务含义**：当安装/预埋件安装任务的"通知可启动/通知启动"节点激活时，如果关联了施工包且施工包状态正常，就自动完成派单员的任务指派，无需派单员手动操作。

## 四、完整决策树（含所有分支）

```
activateTaskDispatch(taskDispatch, materialTask)
  │
  ├─ [入口条件] ─────────────────────────
  │   从 activateTaskDispatch(taskDispatch) 进入的前提：
  │   ├─ processStatus == UN_ACTIVE(1)
  │   ├─ planActivateTime 在 [2000-01-01, 今天结束) 内
  │   └─ processCode 为空
  │
  ├─ 步骤①：processCode 再次检查
  │   └─ 非空? → v2激活, return
  │
  ├─ 步骤②：shouldActivate()
  │   ├─ materialTask 或 taskDispatch 为空? → true
  │   ├─ ACTIVATE_MODE == 0 (PLAN_TIME)?
  │   │   └─ activateScheduleRelation 是 TODAY/AFTER?

  │   │       │   ├─ 状态达标且未取消 → true
  │   │       │   └─ 状态未达标/已取消 → false, return
  │   │       └─ 否 (BEFORE) → true（不过滤）
  │   ├─ ACTIVATE_MODE == 2 (DEPENDENT_NODE)?
  │   │   ├─ 查路由表 → 查前置任务
  │   │   ├─ 无前置任务 → true
  │   │   ├─ 前置有未完成的 → false, return
  │   │   └─ 前置全部完成 → true
  │   └─ 其他模式 (IMMEDIATELY/未配置等) → true
  │
  ├─ 步骤③：checkInterceptConfigure()
  │   ├─ 查尾款拦截配置
  │   ├─ 无配置 → true
  │   ├─ 有配置 + 尾款已结清 → true
  │   └─ 有配置 + 尾款未结清 → false

  │   ├─ processStatus: UN_ACTIVE(1) → UNCOMPLETED(2)
  │   └─ currentNodeTime: old → now

  ├─ 步骤⑤：查第一个 UN_ACTIVE 节点

  │   ├─ 没查到? → 跳过步骤⑥⑦, 直接结束
  │   └─ 查到第一个节点?
  │       ├─ 更新: processStatus → UNCOMPLETED(2)
  │       └─ 更新: startTime → now
  │
  └─ 步骤⑦：激活后置处理
        ├─ completeAssignerTaskWhenActivate()
        │   ├─ 查施工包
        │   ├─ 有施工包 + 安装任务 + 通知节点 + 施工包状态正常
```

## 六、关键设计要点

| 要点 | 说明 |
|------|------|
| **幂等性** | 只在 `processStatus == UN_ACTIVE` 时执行，已激活的不会重复激活 |
| **事务性** | 整个方法在 `@Transactional` 中，激活状态和节点更新在一个事务内 |
| **异步触发** | 本方法本身是同步的，但调用方 `activateTaskDispatchAsync` 是在事务 `afterCommit` 后用 `CompletableFuture.runAsync` 异步执行的 |
| **只激活第一个节点** | 不激活全部节点，只激活 `currentNodeType` 对应的第一个节点，后续节点由 `handleNode` 流转触发 |
| **多条件拦截** | 激活被两重条件拦截：`shouldActivate`（业务条件）和 `checkInterceptConfigure`（财务条件） |
| **v1/v2 共存** | 有 `processCode` 的走 v2（新版流程定义），无的走 v1（旧版模板），本方法是 v1 路径 |
