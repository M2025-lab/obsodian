# 四、**核心能力矩阵：**

## 4.1 agent搭建更简单

### 4.1.1 skill/Tool/知识库可插拔

Agent 本身只是一个"壳"，它的能力全部来自外挂：

- **加一个 Skill**（Markdown 指令文件）→ Agent 学会了怎么处理某类场景
    
- **绑一个 Tool**（本地方法或 MCP 远程接口）→ Agent 就多了一项实操能力，搜订单、算报价、调外部服务
    
- **挂一个知识目录** → Agent 就能回答这个领域的问题
    

![](https://beike.feishu.cn/space/api/box/stream/download/asynccode/?code=YzY1Y2Q5ZTAzNGNmMmMzNmYzOTc5M2YzZTQ2NzlhM2RfRERTRkNxVno0YjJ3bnJwbmk0QzdWN3FnQ3NIS3lFRUhfVG9rZW46Qkw0UGJ5Vllzb0J5Y1V4aEVVZWNkaUZtbjJkXzE3ODQxOTYxMTI6MTc4NDE5OTcxMl9WNA&add_watermark=true&scene_type=CCM)

**目前成果**：

平台已沉淀：**31 个 Skill 领域，85 个 Tool 方法**（系统内嵌 62 个 / MCP 接口 23 个），全部可组合复用。新增一个 智能问答 Agent？选几个 Skill、勾几个 Tool、挂上供应链知识库，完成。不需要改一行 Java 代码。

**这样的优势是：**

- **零代码极速对接**：工具对接周期从周级缩短至分钟级，无需编写任何集成代码
    
- **全栈****资源可插拔**：通用工具、业务工具、知识库全支持动态配置，线上零风险
    
- **积木式能力拼装**：资源独立解耦，可自由组合生成不同场景 AI 能力
    
- **全域资产复用**：一次注册，全平台所有 Agent 共享，彻底消除重复开发
    

  

### 4.1.2 空间隔离，一键上线

平台支持三个粒度的配置同步，覆盖从整体上线到局部调优的全部场景：

![](https://beike.feishu.cn/space/api/box/stream/download/asynccode/?code=Y2ZiODU1NGNiODhjMzI3N2JjZjhhZTAwNTk5YzI1YTVfSDlRbm56V1hyQ0xldmE0MTdWZ3BCcXFJUTBqd3F2S2hfVG9rZW46VnJKUWJrYkw1b2pOeUN4eDJVWmNVVnlSbm9oXzE3ODQxOTYxMTI6MTc4NDE5OTcxMl9WNA&add_watermark=true&scene_type=CCM)

- **环境****同步 — 批量晋升，整体上线**
    

将当前环境的全部空间和配置一次性同步到目标环境，适合完整版本的上线流程。开发调好、测试通过，点一下直接推生产

- **空间****同步 — 按需迁移，快速复用**
    

将指定空间的配置（渠道、Agent、Prompt、技能、知识库）同步到另一个空间，可跨环境也可同环境内复制。适合新业务线冷启动——直接从成熟空间克隆一份，改改配置就能跑。

- **Agent 同步 — 精准同步，局部调优**
    

单个 Agent 粒度，不受空间限制。Prompt 改了、技能脚本更新了，只同步这一个 Agent，不影响其他配置。适合快速迭代期的小步调整。

#### **设计亮点1：LLM 只生成元数据，不重写原文**

切分后每个 Section 的内容原样保留，LLM 仅负责生成 summary（摘要）和 scenarios（检索短语）。规避了 LLM 重写原文带来的幻觉风险，保证 Agent 读到的是真实内容。

  

#### **设计亮点2：list_knowledge_index-两阶段****混合检索**

**第一阶段——两路****并行****粗筛，给 Agent 一份精简的"知识目录"：**

- **关键词评分路**（本地、零成本）：对 query 分词后，按字段权重评分（适用×3、描述×2、摘要×1），取 Top-20 返回。三层安全网兜底：条目数 ≤20 时全量返回，命中数 <10 时自动补至 10 条。
    
- **ES 语义****向量****路**（KNN + match_phrase + RRF 融合）：补充关键词无法命中的语义相关内容，两路结果去重合并。
    

**第二阶段——LLM 看目录，按****内容类型****分流精读：**

|   |   |   |
|---|---|---|
|**类型**|**检索方式**|**适用场景**|
|知识文档|读原文切片，LLM 综合理解|施工规范、操作手册|
|QA 表格|行级向量搜索，双阈值置信度（≥0.98 直接返回，0.78~0.98 返回候选，<0.78 过滤）|标准问答、FAQ|
|Excel 表格|关键词行匹配（含合并单元格感知）|数据表、配置表|

  

```Plain
    对于 Q&A 类表格文档（操作手册、FAQ），list_knowledge_index 会标注条目类型为「表格」，Agent 后续调用 search_excel_rows 做精确行匹配，而非向量检索——关键词比向量更适合结构化表格查询。
```

## 4.4 **Agent 持续运营：从"能跑"到"能查、能改、能优化"**

**核心痛点**

- 线上出问题查得到：全链路 Trace，一步定位是模型错了还是工具错了
    
- 版本迭代测得准：自动打分对比，不用人工一条条验
    
- 用户投诉追得到：按用户 ID 查历史对话，快速复盘
    

  

以下是我们围绕这三个问题构建的完整运营体系。

### **4.4.1 全链路可观测性**

基于 Langfuse 构建三层追踪覆盖，从请求入口到 LLM 调用再到工具执行，每一步都可追踪：

|   |   |   |
|---|---|---|
|**层级**|**追踪对象**|**能力**|
|请求层|用户 ID、会话 ID、运行环境|支持按用户、会话、环境多维度过滤查询|
|LLM 调用层|每次 ChatModel 的提示词、回复、Token 消耗、响应延迟|每条 LLM 调用生成独立追踪链路，包含完整输入输出|
|工具调用层|每次 MCP/内部工具的名称、参数、返回值、耗时、成功/失败状态|嵌套在 LLM 调用 Span 下，最终聚合到根 Span|

### **4.4.2 自动化评测体系**

在线 Tracing 解决的是「看」，但不够。每次调整 Prompt、修改 Skill、更换模型后，Agent 的回答质量变好还是变坏，靠人工一条条测不现实。我们构建了一套完整的自动化测评闭环：

- **自动打分**：对话结束后，Judge 机器人异步评分（0~5 分），输入包含用户问题、工具调用链路和 Agent 回答，从相关性、准确性、完整性、逻辑性、流畅性综合评估，输出评分 + 评分原因，定时同步回写到业务数据库
    
- **分级标注**：每条评分同时标注回答类别——好回答（4~5 分）/ 有效回答（3~5 分）/ 无效回答（0~2 分），便于快速分层筛选和定位问题
    
- **批量对比**：新版本 Prompt/Skill 发布前，用历史评测集跑一遍，自动对比分数变化
    
- **退步高亮**：对比报告中自动标记分数下降的 case，聚焦问题条目

## 4.5 **长期记忆：Agent 越用越懂你**

一个真正"智能"的 Agent，不应该每次对话都从零开始。长期记忆系统让 Agent 在多次对话后逐步"认识"用户——记住偏好、积累经验、感知行为规律。

![](https://beike.feishu.cn/space/api/box/stream/download/asynccode/?code=NjUwMTQ1YjA3YzM0MDgwOGNiMjc5N2ZkNTVkZWNmN2JfbGYyYVJ2R3Y0VFgyTkNuRGZLa21yZkFiV1A2VUtKOG1fVG9rZW46WFIzaGJwN2xKb05jWm94M1RmaWNacW1PbnJkXzE3ODQxOTYxMTI6MTc4NDE5OTcxMl9WNA&add_watermark=true&scene_type=CCM)

  

### 4.5.1 三类记忆，各司其职

|   |   |   |
|---|---|---|
|**记忆类型**|**存什么**|**举例**|
|**偏好**|交互风格和习惯|"喜欢先看异常工地和延期预警""偏好简洁回答，不喜欢大段文字"|
|**经验**|讨论过的决策和解法|"2026-06 讨论过工期延期根因分析方法""复盘过精装修项目的验收流程优化"|
|**行为模式**|高频关注什么|"近 30 天查询延期预警 45 次""习惯在周一早上查看工地概览"|

  

提取逻辑克制——对话轮数 ≥3 且用户发言 >100 字时才触发，不对每条消息都调用大模型，成本可控。

  

### 4.5.2 记忆如何生效

每次新会话开始时，系统加载该用户的记忆快照，整次会话内容保持不变（避免对话中途记忆变化导致前后不一致）。会话结束后如果提取了新记忆，旧快照自动失效，下次新会话加载最新版本。

记忆注入到系统提示词的开头，500 字以内，按优先级排列：偏好全量 > 行为模式全量 > 经验取前 5 条。

  

### 4.5.3 角色专属提取

不同角色关注的信息完全不同，系统按角色使用不同的提取规则，精准度远高于通用方式。

举例——项目经理专属提取规则：

- **偏好**：回答详细程度（简短指导 or 分步骤讲解）、先看哪类信息（异常工地 / 延期预警 / 整体概览）
    
- **经验**：处理过的施工难题、有效的工期和排产管理方法、与设计师和业主的协调经验
    
- **习惯**：常关注的问题类型（材料进场 / 工期节点 / 验收标准 / 业主沟通）
    
      
    

> **下半年规划①**：引入记忆去重机制——写入新记忆前，先检索已有记忆内容，由大模型判断该条应该新增、更新还是跳过，防止同一信息反复写入，让记忆越来越准而不是越来越多。
> 
> **下半年规划②**：接入角色画像——对每个角色已有现成的用户档案，接入后可直接作为记忆初始值写入，新用户第一次对话起就有基础画像，不再需要从零积累。
> 
> **下半年规划③**：记忆检索——当用户积累了大量偏好、经验、行为记录后，现有的"全量加载 + 按优先级截断"方式会失效：500+字预算装不下，重要记忆反而被截掉。规划引入按问题语义检索记忆的机制，每次对话时根据用户当前问题，从记忆库中召回最相关的那部分，而不是固定加载前 N 条，让记忆真正做到按需供给、越积越精准。

  

**长期记忆让 Agent 从"工具"变成"伙伴"——第一次对话你可能需要解释背景，第二次它已经知道你是工长、管着 20 个工地、最关心延期预警。**

  

# 五、与主流框架的对比

目前市面上的 AI Agent 框架大致分为四种路线：

|路线|代表框架|核心理念|
|---|---|---|
|企业级图编排|LangGraph|状态机驱动，精细控制执行流|
|多角色协作|CrewAI、AutoGen|多 Agent 像团队一样协作|
|低代码全栈平台|Dify|拖拽构建，业务人员可直接操作|
|自改进个人 Agent|Hermes|越用越聪明，自动积累技能|

Construction Agent 基于 Spring AI Alibaba 构建，面向家装垂直领域，走的是垂直深耕 + 运营友好路线，和上述框架有明显的差异点。

我们的差异在哪里：**通用框架解决"能跑起来"的问题，我们解决"能持续运营"的问题。**

LangGraph、CrewAI 这类框架解决的是 Agent 如何编排、如何协作，但不关心：

- Prompt 改了怎么不重启上线
    
- 多个业务线怎么互不干扰
    
- 测试环境调好的配置怎么快速推到线上
    
- 低质量回答怎么自动发现、诊断、修复
    
- Agent 怎么越用越了解用户
    

这些是纯粹的工程框架留给开发者自己解决的问题，而在家装业务场景下，这些问题每周都会发生。

我们在 Spring AI Alibaba 的基础上，围绕"可运营"这个目标，建了一套完整的配套体系：空间隔离保证多线并行、版本管理支撑快速迭代、一键同步打通环境晋升、调优工作台闭环持续优化、长期记忆让 Agent 越用越聪明。

# 七、**H2 规划：从"能用"到"好用、可信、会学习"**

当前平台已在生产环境运行，具备多 Agent 路由、知识库检索、工具调用等核心能力。Q3 目标是补齐持续运营和智能进化的底层能力。

**核心问题**：当前 Agent 上线即"黑盒"，出问题靠用户反馈，能力改进靠人工，缺乏研发到运营的完整闭环。

**H2 目标**：构建 Agent 从研发、上线到持续迭代的完整工程体系。
