## 一、实验概述

  

### 1.1 实验目的

  

对比四种不同的知识库检索与生成技术方案在领域知识问答上的表现，评估各方案在事实准确性、答案完整性、召回率和综合评分四个维度上的优劣。

  

### 1.2 实验设计

  

- **测试集**：70 道 BSU 领域问答题目，覆盖 7 个方向，每个方向 10 题
    
- **知识库**：11 份 BSU 体系核心文档（.docx 格式）
    
- **生成模型**：统一使用 qwen3.6-plus 作为回答生成模型（除 Ours 方案使用内部 Agent 体系）
    
- **评测模型**：GPT-5.5 作为独立裁判，对生成答案进行多维度打分
    
- **评测指标**：事实准确性(1-5)、答案完整性(1-5)、召回率(0-1)、综合评分(1-10)

## 四、四种技术方案介绍

  

### 4.1 方案A：标准 RAG 检索 （Bella）

  

**技术架构**：

  

```Plain
用户问题 → RAG API 语义检索 → Top-5 召回(score>0.6) → qwen3.6-plus 生成回答
```

  

**核心参数**：

- 检索接口：`https://openapi-ait.ke.com/v1/api/rag/retrieval`
    
- 检索模式：`semantic`（语义向量检索）
    
- Top-K：5
    
- 分数阈值：score > 0.6
    
- 文件范围：10 个预上传的 file_id
    
- 生成模型：qwen3.6-plus
    

**实现位置**：`TestRunnerRag.java`

  

**特点**：

- 经典 RAG 架构，先检索后生成
    
- 基于向量语义相似度匹配
    
- 检索结果按片段拼接后作为 LLM 上下文
    

---

  

### 4.2 方案B：LightRAG 图增强检索（GraphRAG）

  

**技术架构**：

  

```Plain
11份 .docx 文档 → LightRAG 索引构建(实体抽取+关系提取+图谱构建) → Hybrid 混合检索(向量+图) → qwen3.6-plus 生成回答
```

  

**核心参数**：

- LightRAG 版本：lightrag-hku
    
- 索引方式：自动实体抽取 + 关系提取 + 知识图谱构建
    
- 检索模式：`hybrid`（混合本地/全局图谱检索 + 向量检索）
    
- LLM 模型：qwen3.6-plus（用于实体抽取、关键词提取、回答生成）
    
- Embedding 模型：baai-bge-1b-large-v4-embedding-20240815
    
- 自定义 Prompt：使用中文优化提示词（`prompt.py`）
    

**实现位置**：`tests/lightRAG/run_benchmark.py` + `lightrag_demo.py`

  

**特点**：

- 轻量级知识图谱方案，自动从文档中提取实体和关系
    
- 支持 4 种检索模式：naive（纯向量）、local（局部图）、global（全局图）、hybrid（混合）
    
- 实验使用 hybrid 模式，结合图谱遍历和向量检索
    
- 构建向量与图谱耗时较长，约 1 小时
    
- 知识图谱可捕获文档间的隐性关联
    

---

  

### 4.3 方案C：Elasticsearch 混合检索（ES-Hybrid）

  

**技术架构**：

  

```Plain
11份 .docx → 文本提取(段落+表格Markdown) → ES 上传(raw content) → KNN+Match_Phrase+RRF 混合检索 → Top-10 → qwen3.6-plus 生成回答
```

  

**核心参数**：

- 搜索引擎：Elasticsearch
    
- 文档预处理：docx 文本提取（段落识别 + 表格转 Markdown + docx 标题层级识别）
    
- 检索方式：KNN（k-近邻向量检索）+ match_phrase（短语匹配）+ RRF（倒数排名融合）
    
- Top-K：10
    
- 生成模型：qwen3.6-plus
    

**实现位置**：`ESHybridTest.java`

  

**特点**：

- 结合向量检索和关键词精确匹配
    
- RRF 融合算法综合两种检索结果排序
    
- docx 预处理保留文档结构信息（标题层级、表格格式）
    
- 相比纯语义检索，对精确术语匹配更友好
    
- ES 索引全量文档，检索时返回完整片段
    

---

  

### 4.4 方案D：分片构建文本索引 + 本地文件检索 （类 LLM WIKI）

**技术架构**：

  

```Plain
用户问题 → BsuSubAgent → list_knowledge_index 获取知识索引 → 
fetch_knowledge_doc 读取相关文档 → search_excel_rows 精确表格搜索 → 
Agent 多步推理 → 生成回答
```

  

**核心****组件**：

1. **KnowledgeTool**（`KnowledgeTool.java`）：
    
    1. `list_knowledge_index(query)`：返回与查询相关的知识文档索引
        
    2. `fetch_knowledge_doc(filePath)`：获取指定文档完整内容（限 20000 字）
        
    3. `search_excel_rows(filePath, query)`：在 Excel/MD 表格中按关键词搜索行
        
2. **知识****索引****服务**（`KnowledgeIndexService`）：
    
    1. 每份文档预生成 `_index.md` 索引文件
        
    2. 支持按目录组织知识源配置
        
3. **Agent** **编排**：
    
    1. `BsuSubAgent`：BSU 领域专用 Agent
        
    2. Agent 自主决定调用哪些知识工具
        
    3. 支持多步推理：先查索引定位文档 → 再获取文档内容 → 如需要再搜索表格
        

**实现位置**：`TestRunner.java`（调用 BsuSubAgent）

  

**特点**：

- Agent 自主决策调用哪些工具和顺序
    
- 先通过索引缩小范围，再精确检索
    
- 工具粒度细（索引搜索、完整文档读取、表格搜索各司其职）
    
- 知识库以 Markdown 格式组织，支持结构化表格查询
    
- 不需要预构建向量索引或知识图谱

## 五、评测方法

  

### 5.1 评分标准

  

使用 GPT-5.5 作为独立评测裁判模型，对每组答案按以下 4 个维度打分：

  

|   |   |   |
|---|---|---|
|维度|分值范围|评分标准|
|事实准确性|1-5|回答中的事实是否与标准答案一致，是否有错误信息|
|答案完整性|1-5|是否覆盖标准答案的所有关键信息点|
|召回率|0-1|知识库中相关信息的覆盖比例|
|综合评分|1-10|综合考虑准确性、完整性、清晰度后的整体评价|

  

### 5.2 评测流程

  

```Plain
各方案生成 answers → 保存至 result-{方案}.xlsx → ScoringRunner 逐题调用 GPT-5.5 打分 → 保存至 scored_result-{方案}.xlsx
```

`ScoringRunner.java` 配置：

- 裁判模型：GPT-5.5 （使用高推理能力模型进行）
    
- Temperature：0.1（确保评分一致性）
    
- Max Tokens：2000
    
- 每个维度单独打分，包含详细评语
    
- 支持断点续评和失败重试

- **ES-Hybrid 表现中等**，混合检索有改善但仍不及图增强方案

## 七、详细分析

  

### 7.1 各方案优势场景

  

**RAG 的优势**：

- 简单、直接的事实性查询（如 #2 "BSU包含哪三个核心机制？" 得 10 分）
    
- 标准答案与文档原文高度一致时的精确匹配
    

**GraphRAG 的优势**：

- 条件判断场景（"什么情况下...？"），图谱关系有助于区分不同条件分支
    
- 鲁棒性/对抗测试，口语化表达通过实体链接找到正确知识
    
- 跨文档概念关联，知识图谱自动捕获文档间的隐式关系
    
- 但偶尔会引入"幻觉"——将不同文档段落错误关联
    

**ES-Hybrid 的优势**：

- 对包含精确术语的查询（如具体费率、城市名）
    
- RRF 融合可弥补纯语义检索的不足
    
- 但表格数据提取方向表现不佳，暴露出结构化检索的短板
    

**Ours 的优势**：

- 表格数据提取最强（search_excel_rows 精确行级搜索）
    
- 跨文档推理（Agent 多步调用 list → fetch → search）
    
- Agent 可自主判断何时需要更精确的工具
    
- 检索能力综合评分最高
    

### 7.2 关键差异分析

  

1. **图增强 vs Agent 多步调用**：
    
    1. GraphRAG 通过图谱自动建立跨文档关联，在条件推理上更自然
        
    2. Ours 通过 Agent 工具编排实现多步检索，在精确数据提取上更强
        
    3. 两者在不同维度互补：GraphRAG 召回更好，Ours 答案更准确完整
        
2. **表格检索能力**：
    
    1. Ours 的 search_excel_rows 在表格方向以 7.60 分大幅领先 ES-Hybrid 的 4.90
        
    2. 说明专用表格搜索工具对结构化知识库的重要性
        
    3. RAG 和 GraphRAG 依赖切片质量，表格被截断时效果变差
        
3. **跨文档推理**：
    
    1. Ours（7.60）> GraphRAG（7.00）> ES-Hybrid（6.60）> RAG（4.90）
        
    2. 跨文档推理需要建立文档间的关联路径，RAG 仅靠向量相似度完全不够
        
    3. Agent 的多步调用和图谱的关系遍历都比单步检索有效
        

### 7.3 ES-Hybrid 表格方向异常分析

  

ES-Hybrid 在表格数据提取方向得分仅 4.90，是所有方案中最低的。原因分析：

  

- ES RRF 融合在精确关键词匹配和语义搜索之间可能存在信号干扰
    
- docx 表格转 MD 存在格式丢失（合并单元格处理不完善）
    
- ES 全文索引对表格行级搜索不如 Ours 的 `search_excel_rows` 精确
    
- 表格方向的"精确匹配"需求与 ES 的"相似度排序"存在根本性偏差

## 八、结论与建议

  

### 8.1 核心结论
