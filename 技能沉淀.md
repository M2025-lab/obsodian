# 如果你是 Java 开发，我建议把 `GROUP BY` 理解成 **Java 的 Map 分组统计**，这样最好理解。

---

# 一、先理解 GROUP BY 到底在干什么

假设有一张工人表：

|项目|工种|工人|
|---|---|---|
|A|瓦工|张三|
|A|瓦工|李四|
|A|木工|王五|
|B|瓦工|赵六|

如果写：

```sql
SELECT *
FROM worker;
```

结果就是原始数据。

---

如果写：

```sql
SELECT worker_type
FROM worker
GROUP BY worker_type;
```

得到

|工种|
|---|
|瓦工|
|木工|

为什么？

因为 `GROUP BY` 把相同工种放到了一组。

可以想象成：

```
瓦工
 ├── 张三
 ├── 李四
 └── 赵六

木工
 └── 王五
```

---

Java 中就相当于：

```java
Map<String, List<Worker>> map =
workers.stream()
.collect(Collectors.groupingBy(Worker::getType));
```

---

# 二、GROUP BY 后，SELECT 能写什么？

这是**第一大坑，也是面试最喜欢问的。**

假设：

```sql
SELECT
    worker_type,
    worker_name
FROM worker
GROUP BY worker_type;
```

很多数据库（如 MySQL 开启宽松模式）可能能执行，但 Hive、Spark SQL 等一般会报错。

为什么？

因为：

```
瓦工
 ├── 张三
 ├── 李四
 └── 赵六
```

数据库会问：

> **worker_name 到底返回谁？**

是张三？

还是李四？

还是赵六？

它不知道。

所以，**GROUP BY 以后，SELECT 中只能出现两类字段：**

1. **分组字段**
    

例如：

```sql
GROUP BY worker_type
```

那么

```sql
SELECT worker_type
```

可以。

---

2. **聚合函数**
    

例如：

```sql
COUNT(*)
SUM(price)
MAX(score)
MIN(score)
AVG(score)
COLLECT_SET(name)   -- Hive
```

例如：

```sql
SELECT
    worker_type,
    COUNT(*)
FROM worker
GROUP BY worker_type;
```

结果：

|工种|人数|
|---|---|
|瓦工|3|
|木工|1|

---

# 三、GROUP BY 会改变数据粒度

这是很多新手容易忽略的。

原始数据：

|项目|工人|
|---|---|
|A|张三|
|A|李四|
|B|王五|

一共有 **3 行**。

如果：

```sql
GROUP BY project
```

变成：

|项目|
|---|
|A|
|B|

只剩 **2 行**。

也就是说：

> **GROUP BY 会把多行压缩成一行。**

因此，很多明细数据会丢失。

---

# 四、聚合函数对 NULL 的处理

例如：

|工资|
|---|
|100|
|NULL|
|200|

执行：

```sql
SELECT SUM(salary)
FROM worker;
```

结果：

```
300
```

因为：

**SUM、AVG、MAX、MIN 都会忽略 NULL。**

但是：

```sql
COUNT(*)
```

统计的是所有行。

结果：

```
3
```

而：

```sql
COUNT(salary)
```

只统计非 NULL。

结果：

```
2
```

这是线上非常容易踩坑的地方。

---

# 五、GROUP BY 和 DISTINCT 的区别

很多人以为：

```sql
GROUP BY worker_type
```

和：

```sql
SELECT DISTINCT worker_type
```

一样。

实际上：

如果只是去重，确实效果一样。

例如：

```sql
SELECT DISTINCT worker_type
FROM worker;
```

和：

```sql
SELECT worker_type
FROM worker
GROUP BY worker_type;
```

结果一样。

但是：

只有 `GROUP BY` 才能做统计。

例如：

```sql
SELECT
    worker_type,
    COUNT(*)
FROM worker
GROUP BY worker_type;
```

DISTINCT 做不到。

---

# 六、多字段 GROUP BY

例如：

```sql
GROUP BY
project_order_id,
worker_type
```

表示：

**项目和工种都相同才算一组。**

例如：

|项目|工种|
|---|---|
|A|瓦工|
|A|木工|
|B|瓦工|

实际上分成：

```
(A,瓦工)

(A,木工)

(B,瓦工)
```

三组。

不是先按项目，再按工种。

而是：

> **把多个字段组合起来作为分组键。**

Java 中相当于：

```java
Map<Pair<Project, WorkerType>, List<Worker>>
```

---

# 七、GROUP BY 后 JOIN 要小心

这是公司里最容易出问题的地方。

例如：

订单：

|订单|金额|
|---|---|
|1|100|
|2|200|

商品：

|订单|商品|
|---|---|
|1|A|
|1|B|
|2|C|

如果：

```sql
SELECT
    order_id,
    SUM(price)
FROM order
JOIN item
GROUP BY order_id
```

由于：

订单1有两个商品。

JOIN 后：

```
订单1
订单1
订单2
```

金额：

```
100
100
200
```

最后：

```
SUM(price)=400
```

而正确应该：

```
300
```

所以：

> **JOIN 后 GROUP BY，一定要先确认 JOIN 是否导致数据膨胀（数据重复）。**

这是数据开发中非常经典的问题。

---

# 八、Hive 中 GROUP BY 很耗性能

Hive 的 `GROUP BY` 通常意味着：

- 数据需要重新分区（Shuffle）
    
- 相同 key 的数据要发送到同一个 Reducer
    
- 数据量大时容易产生数据倾斜（Skew）
    

例如：

```sql
GROUP BY city
```

如果：

```
北京 9000万条

上海 100万条
```

那么处理北京的那个 Reducer 会特别慢，这就是**数据倾斜**。

所以大表 `GROUP BY` 要特别关注性能。

---

# 九、结合你最近看的 SQL

你之前发过：

```sql
SELECT
    project_order_id,
    worker_c_ucid,
    MAX(worker_type_name) AS worker_type_name
FROM tmp3
WHERE worker_type_name NOT IN ('产业工人','总工工人')
GROUP BY
    project_order_id,
    worker_c_ucid;
```

这里为什么可以用 `MAX(worker_type_name)`？

因为：

对于同一个：

```
(project_order_id, worker_c_ucid)
```

可能有多条记录。

例如：

|项目|工号|工种|
|---|---|---|
|A|1001|瓦工|
|A|1001|产业工人|

作者想保留一个工种。

由于前面已经：

```sql
WHERE worker_type_name NOT IN (...)
```

过滤掉了：

```
产业工人

总工工人
```

剩下：

```
瓦工
```

于是：

```sql
MAX(worker_type_name)
```

实际上只是**随便取一个值**。

很多公司都会这样写。

# 最后给你总结一个工作中最重要的口诀

每次写 `GROUP BY`，都问自己这五个问题：

1. **我是按什么维度分组？**（例如项目、工人、日期）
    
2. **`SELECT` 中有没有字段既不在 `GROUP BY`，又没有使用聚合函数？**（这是语法和逻辑风险）
    
3. **`JOIN` 后数据有没有变多？会不会导致统计翻倍？**（最容易出线上问题）
    
4. **聚合函数遇到 `NULL` 会不会影响结果？**（特别是 `COUNT(col)` 与 `COUNT(*)` 的区别）
    
5. **这次 `GROUP BY` 之后，数据粒度变成什么了？**（例如从"一行一个工人"变成"一行一个项目"）
    

如果你掌握了这五点，已经能避免 **80% 以上** 的 `GROUP BY` 使用错误。
